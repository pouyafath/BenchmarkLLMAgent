```

          ↑                            ↑                ↑
```

### Why This Breaks LLM Processing

1. **Attention mechanism limitation**: LLMs have better attention to beginning/end of prompts
   - Beginning: SYSTEM_PROMPT + task header are read carefully
   - Middle: 56KB of code (BEFORE/AFTER) gets less attention
   - End: Instructions at position 59040 (last 82 chars) are read

2. **Token optimization**: LLM likely uses KV cache and attention patterns optimized for:
   - System instructions (read fully)
   - Task description (read fully)
   - Long context (sampled/compressed)

3. **Generative bias**: LLM has priors from training data
   - When LLM sees "instructlab" and "training", it generates patches for `instructlab/train.py`
   - This is hallucination based on training data, not following instructions
   - The explicit BEFORE/AFTER content never "overrides" the hallucination

## Lessons for LLM Prompt Engineering

1. **Beware the middle**: Content in the middle of long prompts may not be fully processed

2. **Attention flow**: Optimize for LLM attention patterns:
   - Put critical content at START (right after system prompt)
   - Instructions at END
   - Never bury important content in the middle

3. **Length matters**: 59KB prompts are risky
   - Better: Multiple shorter prompts or iterative approach
   - Sweet spot: 10-20KB total including context

4. **Validation is critical**: Structural validation ≠ semantic correctness
   - A patch can look valid but be completely wrong
   - Real-world testing (harness) is essential

5. **Hallucination beats instructions**: When LLM's training data conflicts with instructions, training data wins
   - Especially with long, ambiguous prompts
   - Use clear structure to prevent hallucination
